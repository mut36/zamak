# 튜닝용 샘플 자막

청크 크기·동시성 계산(`docs/tuning/gemini-limits.md`)의 실측 입력으로 쓰는 자막 파일 자리.

**`.srt` 파일은 gitignore된다.** 자막은 제3자 저작물이라 저장소에 커밋하지 않는다.
빈 파일만 커밋돼 있으니 로컬에서 내용을 채워 쓸 것.

| 파일 | 용도 | 권장 규격 |
|---|---|---|
| `full-movie.srt` | 메인 측정 대상 | 장편 영화 한 편, 블록 1,200~1,600개 |
| `drama-episode.srt` | 대사 밀도 비교군 | 드라마 1화. 영화보다 대사가 촘촘한 케이스 |
| `short-smoke.srt` | 빠른 연기 테스트 | 40~60블록. 청크 1~2개로 끝나는 크기 |

## 고를 때 주의할 점

측정값이 실제 사용 패턴을 대표해야 하므로:

- **원어는 영어** — 현재 도착어가 한국어뿐이라 영→한 팽창률(r)이 핵심 계수다
- **자동생성 자막 피하기** — 문장부호 없고 블록 분할이 기계적이라 토큰 통계가 왜곡된다
- **대사 밀도가 극단적이지 않은 작품** — 다큐/법정물처럼 말이 쉼 없는 것, 액션물처럼 대사가 드문 것 둘 다 평균에서 벗어난다. 표준 케이스 하나 + 밀도 높은 케이스 하나로 p95를 잡는 게 목적
- **HTML 태그(`<i>` 등)가 섞인 파일이 오히려 좋다** — 실사용에 흔하고 토큰을 먹는다

## 측정하는 값

`docs/tuning/gemini-limits.md`의 A(문서 조회)가 채워진 뒤 여기서 B를 뽑는다.

- `t_in` — 블록당 평균 입력 토큰 (번호 + 원문 대사)
- `t_out` — 블록당 평균 출력 토큰 (번호 + 한국어 번역)
- `r = t_out / t_in` — 영→한 팽창률
- 블록 토큰 수의 **p95** — 평균이 아니라 이 값으로 청크 크기를 잡아야 밀도 높은 구간에서 출력 상한을 안 뚫는다
- `P_fixed` — 청크마다 반복되는 고정 프롬프트 토큰
