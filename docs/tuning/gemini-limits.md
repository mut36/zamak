## 3. 반드시 확인할 애매한 지점

계산 결과가 통째로 뒤집히는 항목들이라 값보다 이 정의가 더 중요하다.

| 질문                                               | 답            | 메모                                                                                  |
| -------------------------------------------------- | ------------- | ------------------------------------------------------------------------------------- |
| **TPM은 입력만인가, 입력+출력 합산인가**           | 입력만        | 합산이면 K 상한이 대략 절반이 된다                                                    |
| TPM에 캐시 적중된 입력 토큰도 포함되는가           | 예            |                                                                                       |
| RPM 초과 시 429인가, 큐잉/스로틀인가               | HTTP 429      | 429면 우리는 재시도 안 하므로 그 청크가 원문으로 남는다                               |
| 429에 `Retry-After` 헤더가 오는가                  | 아니오        | 제미나이는 기본 헤더 형태로 명시적인 초 단위 Retry-After를 보장하여 제공하지는 않는다 |
| 무료 등급 한도가 프로젝트 단위인가 API 키 단위인가 | 프로젝트 단위 | BYOK라 사용자마다 별도 예산인지 확인                                                  |

## 5. 컨텍스트 캐싱

`P_fixed`(청크마다 반복되는 고정 프롬프트) 단가를 좌우한다. 캐시가 걸리면 큰 청크의 이점이 줄어든다.

| 항목                                           | 값                      | 메모                                                              |
| ---------------------------------------------- | ----------------------- | ----------------------------------------------------------------- |
| implicit(자동) 캐싱 지원 여부                  | 지원                    | 기본 활성화                                                       |
| implicit 캐시가 걸리는 **최소 접두어 토큰 수** | 4,096                   | 우리 고정 프롬프트가 이보다 짧으면 캐시 자체가 안 걸린다          |
| 캐시 TTL (미사용 시 만료까지)                  | 기본 5분 (수정 가능)    | 청크 간 간격이 이보다 길면 무의미 (명시적 캐싱 사용 시 수정 가능) |
| 캐시 적중을 응답에서 확인하는 필드명           | cachedContentTokenCount | usage metadata의 캐시 토큰 수 필드                                |
| explicit 캐싱 최소 토큰 수                     | 4,096                   |                                                                   |

## 6. thinking / reasoning 설정

자막 번역에는 깊은 사고가 거의 불필요한데, 켜져 있으면 **비용과 출력 여유를 동시에** 잡아먹는다.
단일 최대 절감 레버일 수 있어 별도로 뗀다.

| 항목                                        | 값     | 메모                                                                     |
| ------------------------------------------- | ------ | ------------------------------------------------------------------------ |
| `gemini-3.5-flash`의 thinking 기본값        | medium | 켜짐                                                                     |

### 7-1. 인프라 동시 실행 한도 (조회 필요 — 유료 티어의 진짜 병목)

§2에서 확인됐듯 **제미나이는 동시 요청 수를 따로 막지 않는다.** 유료 티어는 RPM 1,000도 걸리지
않으므로, 동시성 K를 실제로 묶는 건 API가 아니라 **우리 인프라**다. 청크 하나가 함수 하나를
차지한 채 최대 300초 동안 모델 응답을 기다린다.

⚠️ **사용자 1인 기준으로 보면 안 된다.** 동시 접속자 수를 곱한 값이 실제 소요량이다.
1400블록 자막을 B=100으로 처리하면 1인당 함수 14개 → 동시 접속 10명이면 140개.

| 항목 | 값 | 메모 |
|---|---|---|
| 배포 플랜의 동시 함수 실행 한도 (계정/프로젝트당) | | 이 값이 `kmax`이고, 유료 최적 청크 크기를 결정한다 |
| 모델 응답 대기 시간이 함수 실행시간으로 과금되는가 | | 과금되면 대기가 곧 GB-시간 비용 |
| 한도 초과 시 동작 (큐잉 / 429 / 콜드스타트 지연) | | 큐잉이면 시간만, 거부면 번역 실패 |
| 동시 실행 한도를 상향할 수 있는가 / 비용 | | |

## 다음 단계 (이 표가 채워진 뒤)

1. **B 실측** — `samples/`의 자막으로 블록당 입력/출력 토큰(t_in, t_out), 팽창률 r, p95 분산 측정
2. **비용·시간 표** — B를 20/40/100/200/400/800으로 놓고 비용 / 소요시간 / 최악 손실 비교
3. `FREE_CHUNK_SIZE` / `FREE_CONCURRENCY` 확정 후 `app/config/constants.ts`의 TODO 제거

계산 구조 (요약):

```
비용 C(B) = (N/B) × P_fixed × 입력단가          ← B로 줄일 수 있는 유일한 항
          + N × (t_in×입력단가 + t_out×출력단가)  ← B와 무관한 상수

시간 T(B,K) ≈ ceil((N/B)/K) × (TTFT + B×t_out/생성속도)

제약:  B × t_out × (p95 여유계수) ≤ 요청당 최대 출력 토큰
       TTFT + B×t_out/생성속도 ≤ 라우트 타임아웃
       K × 60 / 청크소요초 ≤ RPM
       K × 60 / 청크소요초 × 청크토큰 ≤ TPM
```

**동시성 K는 비용에 영향이 없다** — RPM/TPM 아래에서 시간만 최소화하면 되는 별개 문제다.
