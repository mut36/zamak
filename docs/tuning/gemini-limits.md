# Gemini API 한도 조회표 (수기 입력)

청크 크기(B)와 동시성(K)을 계산으로 정하기 위한 입력값. **추측 금지 — 공식 문서에서 조회한 값만.**
빈칸은 비워두고, 문서에 없거나 애매하면 `?`와 함께 메모를 남길 것.

- 대상 모델(번역): `gemini-3.5-flash` — `app/config/constants.ts`의 `TRANSLATION_MODEL`
- 대상 모델(보조): `gemini-3.1-flash-lite` — `AUX_MODEL` (analyze/enrich/summarize)
- 조회일: 2026-07-20 (§1~6, 대표 조회) / 2026-07-21 (§7-1 인프라, §6 thinking 실측)
- 출처 URL:
  - Gemini rate limits / pricing / model card: 대표가 콘솔·공식 문서에서 직접 조회
  - Vercel Functions 한도(§7-1): <https://vercel.com/docs/functions/limitations>
- **실측으로 확인된 값은 `chunk-size-model.md` §1**에 따로 있다 (이 문서는 *조회*값, 저기는 *측정*값)

## 5. 컨텍스트 캐싱

`P_fixed`(청크마다 반복되는 고정 프롬프트) 단가를 좌우한다. 캐시가 걸리면 큰 청크의 이점이 줄어든다.

| 항목                                           | 값                      | 메모                                                              |
| ---------------------------------------------- | ----------------------- | ----------------------------------------------------------------- |
| implicit(자동) 캐싱 지원 여부                  | 지원                    | 기본 활성화                                                       |
| implicit 캐시가 걸리는 **최소 접두어 토큰 수** | 4,096                   | 우리 고정 프롬프트가 이보다 짧으면 캐시 자체가 안 걸린다          |
| 캐시 TTL (미사용 시 만료까지)                  | 기본 5분 (수정 가능)    | 청크 간 간격이 이보다 길면 무의미 (명시적 캐싱 사용 시 수정 가능) |
| 캐시 적중을 응답에서 확인하는 필드명           | cachedContentTokenCount | usage metadata의 캐시 토큰 수 필드                                |
| explicit 캐싱 최소 토큰 수                     | 4,096                   |                                                                   |

## 6. thinking / reasoning 설정

자막 번역에는 깊은 사고가 거의 불필요한데, 켜져 있으면 **비용과 출력 여유를 동시에** 잡아먹는다.
단일 최대 절감 레버일 수 있어 별도로 뗀다.

| 항목                                        | 값     | 메모                                                                     |
| ------------------------------------------- | ------ | ------------------------------------------------------------------------ |
| `gemini-3.5-flash`의 thinking 기본값        | medium | 켜짐                                                                     |
| thinking을 끄거나 예산을 0으로 둘 수 있는가 | 안됨   | `thinkingBudget: 0`은 무시된다. 먹는 건 `thinkingLevel`                  |
| 최소 thinking 예산 (0 불가라면)             | minimal | `ThinkingLevel.MINIMAL`                                                 |
| 응답에서 thinking 토큰 수를 확인하는 필드명 | `usageMetadata.thoughtsTokenCount` | `[gemini]` 로그의 `thoughts=`                 |

**실측 (2026-07-21) — MINIMAL과 LOW 모두 `thoughts=0`.**

| THINKING_LEVEL | 실측 thoughts/요청 | 표본 |
|---|---|---|
| MINIMAL | 0 | 200블록 청크 |
| LOW | **0** | 200·200·61블록 청크 3개 (전부 0) |
| MEDIUM | 미측정 | |
| HIGH | 미측정 | |

⚠️ 3줄짜리 장난감 파일 1회에서는 LOW가 `thoughts=595`로 나왔으나, **실제 워크로드(200블록)
3회는 전부 0이었다.** 극단적으로 짧은 입력에서만 thinking이 발생하는 것으로 보인다.
결론: **실사용에서 LOW의 thinking 비용은 MINIMAL과 같다(0).** 품질이 더 나은 LOW를 쓸 이유가 있다.

이 실측이 청크 모델의 결론을 바꿨다 — `th=0`이면 "요청마다 붙는 thinking을 줄이려면 B를
키워야 한다"는 논거가 통째로 사라진다. 상세는 `chunk-size-model.md`.

### 7-1. 인프라 동시 실행 한도 — **조회 완료 (2026-07-21), 병목 아님**

출처: <https://vercel.com/docs/functions/limitations> (last_updated 2026-07-01)

| 항목 | 값 | 메모 |
|---|---|---|
| 배포 플랜의 동시 함수 실행 한도 | **Hobby·Pro 30,000까지 자동 스케일** (Ent 100,000+) | 우리가 쓰는 K=14와 3자릿수 차이 |
| 모델 응답 대기가 함수 실행시간으로 과금되는가 | **CPU는 아니오, 메모리는 예** | 아래 주의 참조 |
| 한도 초과 시 동작 | 자동 스케일 구간 내에선 해당 없음 | |
| 함수 실행시간 상한 | Hobby 300초 / Pro 기본 300초·최대 800초 | 우리 `maxDuration=300`은 Hobby에서도 안전 |

**이 절의 원래 전제가 틀렸다.** "청크 하나가 함수 하나를 300초 점유하므로 동시 접속자 수를
곱한 값이 인프라 한도에 걸린다"고 봤지만, Vercel은 30,000까지 자동 스케일한다. 동시 접속
1,000명이 각자 14청크를 돌려도 14,000으로 한도 안이다. **kmax는 유료 티어의 병목이 아니다.**

⚠️ **다만 "대기가 공짜"는 아니다.** Fluid compute의 과금은 두 축이고 성격이 다르다:
- *Active CPU time* — I/O 대기는 **불포함**. Gemini 응답을 기다리는 동안 CPU 과금 없음
- *Provisioned memory time* — 인스턴스가 **살아있는 벽시계 시간**에 비례. 대기도 여기엔 잡힘

즉 대기의 CPU 비용은 0이지만 메모리 비용은 남는다. 그래도 Gemini 토큰 비용(편당 ~$0.15)에
비하면 무시할 수준이라, **B/K 결정에서 인프라 비용은 고려 대상이 아니다.**

### 7-2. 그래서 K를 실제로 묶는 것은 무엇인가

Gemini도(§2) Vercel도(§7-1) 동시성을 막지 않으므로, K는 **제약이 아니라 우리 선택**이다.
유일하게 남는 공유 자원은 **유료 RPM 1,000**이고, 이건 전체 사용자가 나눠 쓴다.

```
사용자 1인이 번역 중 소비하는 RPM = ⌈N/B⌉ ÷ (총 소요초) × 60
동시 번역 가능 인원 ≈ 1,000 ÷ 위 값
```

현재값(B=125, K=14, N=851: 7요청/11초 = 38 RPM)이면 **동시 번역 26명**까지 여유가 있다.
K를 키우거나 B를 줄이면 이 숫자가 줄어든다 — 그게 K의 진짜 트레이드오프다.

## 상태 (2026-07-21)

이 표는 **다 채워졌다.** 남은 미측정은 THINKING_LEVEL MEDIUM/HIGH 품질·비용뿐이고, 그건
B/K 결정에 영향을 주지 않는다(§6 참조 — thinking은 실사용에서 0).

B/K의 최종 결정과 근거는 **`chunk-size-model.md`**에 있다. 이 문서는 그 계산의 입력값이다.

계산 구조 (요약):

```
비용 C(B) = (N/B) × P_fixed × 입력단가          ← B로 줄일 수 있는 유일한 항
          + N × (t_in×입력단가 + t_out×출력단가)  ← B와 무관한 상수

시간 T(B,K) ≈ ceil((N/B)/K) × (TTFT + B×t_out/생성속도)

제약:  B × t_out × (p95 여유계수) ≤ 요청당 최대 출력 토큰
       TTFT + B×t_out/생성속도 ≤ 라우트 타임아웃
       K × 60 / 청크소요초 ≤ RPM
       K × 60 / 청크소요초 × 청크토큰 ≤ TPM
```

**동시성 K는 비용에 영향이 없다** — RPM 아래에서 시간만 최소화하면 되는 별개 문제다.
