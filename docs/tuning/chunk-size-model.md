# 청크 크기 수학모델

`scripts/chunk-model.mjs`가 이 모델의 실행본이다. 기본값은 **전부 실측치**로 갈아끼워져 있다.

```bash
node scripts/chunk-model.mjs                                  # 실측 파라미터
node scripts/chunk-model.mjs N=2000                           # 크레딧 상한 파일
node scripts/chunk-model.mjs file=samples/subtitles/full-movie.srt
```

> **결론 먼저**: `SERVER_CHUNK_SIZE = 500`, `SERVER_CONCURRENCY = 16`(B=500에서도
> 과분하지만 조정 불필요). 무료 티어(150/6)는 **현재 미사용**이다 —
> BYOK가 삭제돼서 전원이 server 티어를 쓴다(`resolveTier()`는 항상 `'server'`).
>
> ⚠️ **2026-07-22, 하루에 세 번 뒤집힌 결론**: ①계산상 B는 정할 수 없다는
> 결론(5절, 상한 두 개뿐 — `B ≤ 3,276`)에 도달해 **B=2000(청킹 없음)**으로 갔다가,
> ②**같은 날 저녁 베타 테스트에서 재번호 드리프트가 실사용으로 재현**돼(8절)
> **B=500**으로 되돌렸다. 지금 값의 근거는 "계산상 최적"이 아니라 "실사용에서
> 사고가 안 났다"는 경험적 안전선이다. 1웨이브 규칙은 여전히 **폐기** 상태(4절).

## 1. 실측 파라미터 (2026-07-21)

461블록 파일 1회 번역의 `[gemini]` 로그 3줄에서 전부 나왔다. 로그 형식:

```
[gemini] model=... thinking=LOW prompt=P cached=0 thoughts=T output=O
```

| 관측 | 블록 | prompt | output |
|---|---|---|---|
| 청크 3 | 61 | 1,942 | 932 |
| 청크 1 | 200 | 3,961 | 3,165 |
| 청크 2 | 200 | 4,090 | 3,383 |

`prompt = P_fixed + blocks × t_in` 연립으로 역산:

| 기호 | 뜻 | **실측** | 이전 추정 | 추정 대비 | 비고 |
|---|---|---|---|---|---|
| t_in | 블록당 입력 토큰 | **15** | 10 | +50% | |
| t_out | 블록당 출력 토큰 | **16** | 20 | −20% | |
| **th** | 요청당 thinking 토큰 | **0** | 100 | −100% | MINIMAL·LOW 모두 0 (gemini-limits §6) |
| P_fixed | 청크마다 반복되는 고정 프롬프트 | **1,028** | 700 | +47% | 작품 notes 포함 |
| v | 출력 생성 속도 | **~220 tok/s** | 120 | +83% | 동시 시작 청크들의 완료시각 차로 역산 (175~244 범위) |
| dens | 최밀 구간/평균 밀도비 (p95) | 1.25 | — | — | 미갱신 (샘플 실측 ~1.2) |
| TTFT | 첫 토큰까지 지연 | 2s (추정) | — | — | 미측정 |

추정이 전부 한 방향으로 틀린 게 아니라는 점이 중요하다 — 입력은 과소평가(t_in +50%,
P_fixed +47%), 출력은 과대평가(t_out −20%, th −100%)였다. 비용 결론이 뒤집힌 건
**출력 쪽 과대평가 때문**이다(3절).

부수 실측: **정렬 실패율 0.22%** (461블록 중 1블록이 원문 유지 — 실패는 청크 1에만
발생, 그 청크 안에서는 1/200 = 0.5%). 이전 측정은 461블록 중 3블록 = 0.65%였으므로
**이번이 더 낮다.** 표본 2개뿐이라 0.2~0.7% 범위로 보는 게 안전하다.

## 2. 공식

```
비용  C(B) = [m·P_fixed + N·t_in]·pin  +  [N·t_out + m·th]·pout
              └ 입력: 반복 프롬프트만 B에 종속 ┘   └ 출력: thinking만 B에 종속 ┘

청크 소요  D(B) = TTFT + (B·t_out + th) / v

동시성     K = min( m, kmax )          ← 4절 참조. RPM 항은 유료에서 비바인딩
총 시간    T(B) = ⌈m/K⌉ · D            (m = ⌈N/B⌉)

제약  B·t_out·dens + th ≤ outcap        (최밀 청크가 출력 상한 안)
      D(B) ≤ timeout                     (청크 하나가 300초 안)
      ↑ 강제되는 건 이 둘뿐이다. "B ≤ 손실 허용치"는 제약이 아니다 — 5-4 참조
```

## 3. th=0이 결론을 뒤집었다

예전 모델의 핵심 주장은 **"비용은 B에 단조 감소하니 되도록 크게"**였고, 그 힘은 전부
`m·th·pout` 항에서 나왔다 — 요청마다 붙는 thinking이 **출력 단가(입력의 6배)**로 과금되니
요청 수(m)를 줄여야 한다는 논리였다.

**`th=0`으로 실측되면서 그 항이 사라졌다.** 남은 B 종속 항은 `m·P_fixed·pin` 하나뿐이고,
입력 단가는 출력의 1/6이라 영향이 작다. 그 결과:

| B | $/편 (N=851) | 총 시간 (K=14) | 최악 손실 |
|---|---|---|---|
| 100 | $0.1556 | 9초 | 100블록 (12%) |
| **125** | **$0.1525** | **11초** | **125블록 (15%)** |
| 150 | $0.1509 | 13초 | 150블록 (18%) |
| 200 (구값) | $0.1494 | 17초 | 200블록 (24%) |
| 851 | $0.1432 | 64초 | 851블록 (100%) |

**B=125와 B=851의 비용 차이는 6.5%**다. 비용이 사실상 평평해진 이상, B는 이제
**다른 축으로 정해야 한다.**

⚠️ 여기서 한때 "시간·손실반경·진행률 셋 다 작은 B를 선호한다"고 결론냈는데, **셋 중
둘이 무효**다 — 시간은 제약이 아니고(4-2), 손실 반경은 B와 무관하다(5-4). 위 표의
"최악 손실" 열도 기댓값이 아니라 **단일 청크 실패 시 최악값**이라는 점에 주의.

## 4. K는 제약이 아니다 — 그리고 유도값도 아니다

### 4-0. K=14에는 근거가 없었다 (2026-07-22 확인)

`git log -S`로 추적한 결과, `CONCURRENCY = 14`는 **최초 커밋 `fb952ea`에서
주석 한 줄과 함께 그대로 들어온 값**이다. ZAMAK 리디자인 이전 코드베이스에서
넘어왔고 유도 과정이 없다.

```
/** Max concurrent chunk translations */
export const CONCURRENCY = 14;      ← fb952ea, 2026-07-17
```

**아래 4-1의 표는 14를 고른 이유가 아니라, 14가 아무것도 위반하지 않는다는
사후 확인이다.** 그리고 §5의 `B ≥ 1500/14 = 107.1`은 그 임의값을 하한의
출발점으로 썼다 — 즉 **B가 K로부터 유도됐고, K는 물려받은 값이었다.**

### 4-1. 무엇도 K를 막지 않는다

`gemini-limits.md` §7-1 조회 결과, **인프라 병목이라는 전제가 틀렸다**:

| 후보 | 실제 값 | 바인딩? |
|---|---|---|
| Gemini 동시 요청 제한 | 없음 (§2) | ✗ |
| Vercel 동시 실행 | 30,000 자동 스케일 (§7-1) | ✗ — 우리 14와 3자릿수 차 |
| Gemini 유료 RPM | 1,000 (전체 사용자 공유) | △ 규모가 커지면 |
| 라우트 타임아웃 300초 | B=851도 64초 | ✗ |
| 출력 상한 65,536 | B=125면 2,500 | ✗ |

즉 **K를 막는 건 아무것도 없다.**

### 4-2. 1웨이브 규칙 폐기 (2026-07-22)

한때 K를 `K ≥ ⌈상한/B⌉`(모든 파일이 1웨이브)로 유도했다. **대표 결정으로 폐기했다** —
2웨이브가 되면 시간이 2배지만 애초에 1분도 안 걸리는 작업이라 무의미하고, 이 규칙은
**K를 B에 묶어놓는 것 외에 아무 일도 안 했다.**

폐기의 결과를 분명히 해둔다: **K도 B도 유도값이 아니다.** 5절이 그 상태를 다룬다.

유일한 실질 예산은 Gemini의 공유 1,000 RPM이고, 소비량은 `rate = K × 60 / D(B)`다 —
**K가 지배하고 B는 D를 통해 간접 영향만 준다.** K=8 기준:

| B | D | 1인당 RPM | 동시 수용 |
|---|---|---|---|
| 50 | 5.6s | 85 | 11명 |
| 125 | 11.1s | 43 | 23명 |
| 200 | 16.5s | 29 | 34명 |
| 500 | 38.4s | 13 | 79명 |

지금 규모에선 전부 무의미한 여유다. 볼륨이 늘어 1,000 RPM이 조여오면 **K를 줄이면
된다** — B를 건드릴 일이 아니다.

## 5. B는 계산으로 못 정한다 (2026-07-22 결론)

여러 번 유도를 시도했고 전부 무너졌다. 이 절은 **왜 무너졌는지**와 **대신 무엇이
필요한지**를 기록한다. 새로 유도를 시도하기 전에 반드시 읽을 것.

### 5-1. 무너진 유도들

| 시도 | 내용 | 무너진 이유 |
|---|---|---|
| "클수록 싸다" | `m·th·pout` 항을 줄이려면 B를 키워야 | **`th=0` 실측** — 항이 통째로 사라짐 (3절) |
| "`⌈1500/14⌉=108` 이상" | K=14 하에서 1웨이브를 만족하는 최소 B | **K=14에 근거가 없었다** (4-0). 상한 1,500도 2,000으로 바뀜 |
| "125엔 실측 기준선이 있다" | 정렬 실패율 3회 측정이 전부 B=125 | **오독.** 0.65%·0.22%는 **B=200**에서 나왔고, 0.07%(B=125)는 프롬프트 개정과 같은 실행이라 교란됨 |
| "1웨이브 규칙" | `K ≥ ⌈상한/B⌉` | **폐기** (4-2) — 시간이 제약이 아님 |

### 5-2. 유도되는 건 상한 두 개뿐

```
출력 상한:  B · t_out · dens ≤ 65,536   →   B ≤ 3,276
타임아웃:   TTFT + B·t_out/v ≤ 300s     →   B ≤ 4,097
```

**둘 다 현재값의 26배 이상 떨어져 있다.** `constants.test.ts`가 이 둘만 강제한다.
계산이 주는 답은 사실상 **"3,276 이하 아무거나"**다.

### 5-3. 나머지 축은 전부 끝점이 없다

| 축 | B에 대한 거동 | 최적점 |
|---|---|---|
| 비용 | 단조 감소, **전 구간 6.5%** | 없음 (최대 B가 답이라 무의미) |
| 시간 | 단조 증가 | 없음 — **그리고 제약이 아님** (4-2) |
| RPM | `K×60/D(B)` — **K가 지배** | 없음 — K로 조절할 일 |
| 진행 링 해상도 | 작을수록 유리 | 없음 (미학) |

### 5-4. ⚠️ 손실 반경은 B와 무관하다

문서와 코드 주석이 오랫동안 **"작은 B가 손실을 줄인다"**고 적었는데 **틀렸다.**

```
기대 미번역 블록 = p · m · B = p · ⌈N/B⌉ · B ≈ p · N
                                                └ B가 소거된다
```

청크 1개 실패가 B블록을 잃는 건 맞지만, **B를 반으로 줄이면 청크 수가 2배가 된다.**
기대 손실은 같다. 작은 B가 바꾸는 건 평균이 아니라 **분산**이다 — "가끔 크게"가
"자주 조금씩"이 될 뿐이다.

이게 결론에 미친 영향이 크다. "시간·손실반경·진행률 3:1로 작은 B가 이긴다"는
집계에서 **시간은 제약에서 빠졌고 손실반경은 무효**다. 남는 건 진행 링 하나뿐이고,
반대편에는 5-5의 품질 축이 있다.

### 5-5. 실제로 B를 가르는 두 양 — 둘 다 미측정, 방향 반대

| 양 | B 증가 시 | 왜 계산이 안 되나 |
|---|---|---|
| **정렬 실패율 f(B)** | **방향 불명** | "청크가 길면 모델이 번호를 더 흐트러뜨리는가" — 경험량. 돌려봐야 안다 |
| **인물 말투 일관성** | **좋아짐** | 청크는 서로를 모른 채 병렬 번역된다. B가 크면 한 청크가 보는 문맥이 넓다 |

두 번째는 `decisions.md`의 `character_voice_registry` 항목이 지목한 **체감 품질 최대
하자**와 같은 문제다. 즉 **작은 B는 그 하자를 악화시킨다.**

### 5-6. ~~B = 2000 (청킹 없음)~~ → B = 500 (2026-07-22 저녁, 재역전)

B=2000을 그날 낮에 도입했다가 **같은 날 저녁 베타 테스트에서 재번호 드리프트가
실사용으로 재현**돼 되돌렸다. 상세는 §8, 결정 기록은 `decisions.md` §2-3-1.

당시 B=2000의 근거였던 것:

- **가장 싸다** (구간 내 최저, `$0.3345`)
- **모델이 영화 전체를 문맥으로 본다** — 5-5의 말투 일관성 축에서 최선

대가로 명시했던 것 중 실제로 문제가 된 건 예상한 항목이 아니었다 — 출력 상한이나
타임아웃이 아니라 **§8의 조용한 재번호 오염**이었다. 대가 표는 참고용으로 남긴다:

| 항목 | B=125 | B=2000 (폐기) | **B=500 (현재)** |
|---|---|---|---|
| 실패 시(청크 전체 API 실패) | 청크 1개(6%)만 원문 | 파일 전체가 원문 | 청크 1개(최대 25%)만 원문 |
| 진행 링 | 16단계 | 시간 추정만 | 최대 4단계 |
| 출력 상한 여유 | 3.8% 사용 | 61% 사용 | 15.3% 사용 |
| 타임아웃 여유 | 4% 사용 | 49% 사용 | 12.8% 사용 |
| 재번호 드리프트 사고 시 오염 범위 | 최대 125블록 | 최대 1,361블록 | **최대 362블록** |

⚠️ **비영어 원어를 받기 전에 `t_out`을 재측정할 것.** 현재 16은 영어 자막 실측값이다.
B=500에서는 밀도가 2배(`t_out≈30`)여도 `500×30×1.25=18,750 < 65,536`이라 출력
상한은 안전하지만, §8의 드리프트 발생률이 원어에 따라 달라질 가능성은 미검증이다.

K=16은 B=500에서도 과분하다(`⌈2000/500⌉=4` ≪ 16, 여전히 1웨이브). 조정 불필요.

### 5-7. 진행률 추정 재유도

`TIMING.FLASH_BATCH_MS = 60초`는 **B와 N에 무관한 상수**였다. 청크가 여러 개일 땐
실제 진행률이 있어서 안 드러났지만, 청크 1개에선 이 추정이 링의 유일한 근거가 된다:

| N | 실제 `D = 2 + N·16/220` | 구 추정 | 오차 |
|---|---|---|---|
| 400 | 31초 | 60초 | −48% |
| 851 | 64초 | 60초 | +6% |
| 1,480 | 110초 | 60초 | +83% |
| **2,000** | **147초** | 60초 | **+146%** |

즉 상한 파일에서 링이 **60초에 99%에 도달해 87초를 더 기어간다.** `estimateTranslationMs()`
(`constants.ts`)로 교체했다 — 청크 모델과 **같은 실측 파라미터**를 쓰므로 재측정 시
둘을 같이 갱신할 것.

값을 바꿔 실험하려면 `NEXT_PUBLIC_CHUNK_SIZE` / `NEXT_PUBLIC_CONCURRENCY`로 덮으면
된다(하네스도 같은 상수를 읽는다). f(B) 비교는 여전히 **미실행**이다.

측정 설계 (프롬프트를 고정해야 교란이 없다 — 지난번 실패 원인):

```bash
NEXT_PUBLIC_CHUNK_SIZE=200 npm run harness -- file=... 
NEXT_PUBLIC_CHUNK_SIZE=125 npm run harness -- file=...
```

같은 파일·같은 프롬프트로 B만 바꿔 **정렬 실패율**을 비교한다(하네스가 이미 측정하는
항목). 말투 일관성은 자동 지표가 없어 **대표 정성 판단**이 필요하다.

### 동적 B를 채택하지 않은 이유

`B = ⌈N/K⌉`로 파일마다 맞추는 안. 1웨이브 규칙이 폐기된 지금은 **애초에 목적이
사라졌다** — 웨이브 수를 맞출 이유가 없다. 게다가 파일마다 B가 달라지면 5-5의
실측을 파일 간 비교할 수 없게 된다.

## 8. 재번호 드리프트 — 조용한 오염 (2026-07-22 발견)

`reassembleTranslatedChunk`(`srt.ts`)가 번호를 신뢰하는 방식의 구멍이다. §2절 공식이나
§5절의 B 트레이드와는 다른 차원의 문제라 별도로 분리한다 — **이건 비용·시간 최적화가
아니라 정합성 버그다.**

### 8-1. 무엇이 잘못됐나

`decisions.md` §2-1에 있던 "타임코드가 밀리는 일은 구조적으로 불가능하다"는 주장이
**틀렸다.** 정확한 보장은 "타임코드가 원본과 다른 값이 되지 않는다"이지 "타임코드와
대사의 짝이 맞는다"가 아니다.

모델이 자막 하나를 둘로 쪼개면(예: 639번의 대사가 길어 "639-앞/639-뒤"로 나눔) 그 뒤
번호를 하나씩 밀려서 매긴다. 밀린 번호(640, 641, …)도 **원본에 실재하고, 아직
안 쓰였고, 이전 번호보다 크므로** 인정 조건 3개를 전부 통과한다. 그 결과:

```
원본 640번(대사 C)의 타임코드에 → 실제로는 639번 뒷부분 번역문이 붙는다
```

번호가 파일 끝까지 하나씩 밀린 채로 재조립되고, `matched` 카운터는 **번호가 붙긴
붙었으므로 100% 성공으로 집계한다.** 기존 실패율 지표(0.5~0.65%)는 "번호가 아예
안 붙은" 경우만 잡으므로 **이 오염은 그 수치에 반영되지 않는다.**

부수 증상: 원본 범위를 벗어난 번호(예: 642)가 나타나면, 그건 재번호의 확실한
증거인데도 지금은 그냥 자막 본문 텍스트로 밀어넣는다 — 화면에 숫자가 대사처럼 뜬다.

### 8-2. 발견 경위

2026-07-22 저녁, 대표가 베타 테스트 중 B=2000 파일에서 실사용으로 재현. 유닛
테스트로도 즉시 재현 확인(4블록짜리 최소 사례, `matched=4/4`인데 639번부터
전부 밀림). 대표 진단: **청크가 600줄 근처를 넘어가면 모델 어텐션이 흐려져
재번호 실수가 는다** — 코드로 증명된 메커니즘은 아니고 경험적 관찰. B=500으로
낮춘 뒤 재발 없음(단기 관찰, `decisions.md` §2-3-1).

### 8-3. 임시 조치와 남은 일

**임시 조치**: B를 500으로 낮춰 사고당 오염 범위를 줄였다(최대 1,361블록 →
362블록). **이건 완화이지 수정이 아니다** — 확률이 0이 됐다는 증거는 없고, 사고가
나면 여전히 조용히 오염된다.

**대표 결정**: 베타 기간엔 B=500으로 운영하며 관찰하고, 재발하면 그때 아래 탐지·복구
로직을 넣는다.

**탐지·복구 초안** (미구현, 재발 시 착수):

1. **탐지**: `indexTranslatedBodies`에서 원본 범위 밖 번호가 나오면 그 시점부터
   "이 청크는 재번호 의심"으로 표시. 지금은 이 신호를 버리고 본문에 섞는다.
2. **복구 — 세 가지 선택지, 우열은 미논의**:
   - (a) 의심 시작 지점부터 그 청크 끝까지 원문 유지. 그 앞은 살린다
   - (b) 청크 전체 원문 유지. 가장 안전하지만 멀쩡한 앞부분도 버림
   - (c) 자동 복구 없이 완료 화면에 경고만 표시, 판단은 사용자에게
3. 어느 쪽이든 **오염 의심 블록을 `unmatched`에 반영**해서 지표가 더 이상
   100%를 거짓 보고하지 않게 할 것.
