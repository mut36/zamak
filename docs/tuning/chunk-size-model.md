# 청크 크기 수학모델

`scripts/chunk-model.mjs`가 이 모델의 실행본이다. 파라미터가 실측되면 거기만 갈아끼우고 재실행한다.

```bash
node scripts/chunk-model.mjs                                  # 현재 추정값
node scripts/chunk-model.mjs N=1400 tout=22 th=150 v=140      # 실측값 반영
node scripts/chunk-model.mjs file=samples/subtitles/full-movie.srt
```

## 변수

| 기호 | 뜻 | 현재값 | 출처 |
|---|---|---|---|
| N | 파일의 자막 블록 수 | 851 / 461 | 샘플 실측 |
| B | 청크 크기 (결정 변수) | — | |
| m | 청크 수 = ⌈N/B⌉ | — | |
| t_in | 블록당 입력 토큰 (번호+대사) | ~10 | 실측 36.5자/블록 ÷ ~4자/토큰 |
| **t_out** | 블록당 출력 토큰 (번호+한국어) | **~20 (추정)** | **로그 output= ÷ 블록수로 실측 필요** |
| **th** | 요청당 thinking 토큰 (MINIMAL) | **~100 (추정)** | **로그 thoughts= 실측 필요** |
| P_fixed | 청크마다 반복되는 고정 프롬프트 | ~700 | 실측 1,385자 |
| dens | 최밀 구간/평균 밀도비 (p95) | 1.25 | 샘플 100블록 창 실측 ~1.2 |
| **v** | 출력 생성 속도 tok/s | **120 (추정)** | **실측 필요: 청크 소요시간에서 역산** |
| TTFT | 첫 토큰까지 지연 | 2s (추정) | |
| **kmax** | 우리가 거는 동시성 상한 | **14 (근거 없음)** | **조회표 §7-1 — 배포 플랜 동시 실행 한도** |
| RPM | 분당 요청 | 무료 15 / 유료 1,000 | 조회표 |
| outcap | 요청당 최대 출력 | 65,536 | 조회표 |
| timeout | 라우트 상한 | 300s | maxDuration |

## 공식

```
비용  C(B) = [m·P_fixed + N·t_in]·pin  +  [N·t_out + m·th]·pout
              └ 입력: 반복 프롬프트만 B에 종속 ┘   └ 출력: thinking만 B에 종속 ┘

청크 소요  D(B) = TTFT + (B·t_out + th) / v

동시성 상한  K = min( m,  kmax,  RPM,  ⌊D·RPM/60⌋ )
                     └ 우리 인프라 ┘  └ 무료 티어의 실질 제약 ┘
총 시간  T(B) = ⌈m/K⌉ · D

제약  B·t_out·dens + th ≤ outcap        (최밀 청크가 출력 상한 안)
      D(B) ≤ timeout                     (청크 하나가 300초 안)
      B ≤ 손실 허용치                     (청크 통실패 = B블록 원문 잔존)
```

핵심 구조: **자막 본문 토큰은 B와 무관한 상수다.** B가 움직이는 건
- 입력에서 `m·P_fixed` (반복 프롬프트) — B↑면 ↓
- 출력에서 `m·th` (요청당 thinking, 출력 단가 6배) — B↑면 ↓
- 시간에서 `⌈m/K⌉·D` — 무료 티어는 RPM이 K를 D/4로 묶어서 **작은 청크의 속도 이점이 상쇄됨**

즉 비용은 B에 단조 감소(클수록 싸다), 시간은 U자형(중간이 최소).

## 티어별로 K의 성격이 다르다

| | K를 정하는 것 | 상태 |
|---|---|---|
| 무료 | 제미나이 RPM 15 — 진짜 제약. `K ≤ ⌊D·15/60⌋` | **확정 (6)** |
| 유료 | 우리 서버리스 동시 실행 한도 `kmax` — 제미나이와 무관 | **미조사** |

제미나이는 동시 요청 수를 따로 막지 않고(조회표 §2) 유료 RPM 1,000도 안 걸린다. 유료에서
K를 묶는 건 **우리 인프라**다 — 청크 하나가 함수 하나를 최대 300초 점유하고, 그 총량은
**동시 접속자 수만큼 곱해진다**. 조회표 §7-1이 이 값의 자리다.

**유료 최적 청크 = N / kmax.** 한 웨이브에 끝나는 가장 작은 청크다. 그보다 크면 웨이브는
그대로 1인데 청크가 길어져 느려지고, 작으면 웨이브가 늘어 느려진다. 검증: kmax=7→B=200,
14→100, 28→50으로 정확히 따라 움직인다.

⚠️ 즉 **유료에는 최적점이 없는 게 아니라, 최적점이 kmax에 못 박혀 있다.** 지금
`SERVER_CONCURRENCY=14`는 최초 커밋부터 실려 온 근거 없는 값이므로, 여기서 나오는
"유료 최적 B=100"도 임의값에 답을 맞춘 것일 뿐이다. 결제 게이트 붙이기 전에 실제 한도를
조회하고 `node scripts/chunk-model.mjs kmax=<실제값>`으로 둘 다 다시 뽑을 것.

## 현재 추정값 기준 결과 (2026-07-20)

무료 티어, 851블록(영화): **최속 B=150 (28초)**. B=25는 4분+, B=40은 1분45초.
무료 티어, 1400블록: **최속 B=200 (36초)**. B=40은 2분51초.
유료 티어(kmax=14 가정): 최속 B=100(20초), 최저비용 B=851($0.277). B=100이 $0.300이라
**최속이면서 최저비용 대비 8% 비쌀 뿐** — kmax가 확정되면 여기가 답이 된다.

민감도: thinking을 5배(500), 속도를 절반(60)으로 흔들어도 **최적점은 100~200 사이에 머문다.**
극단(추정보다 느린 생성 + 긴 출력)에서 B≥600은 TIMEOUT에 걸린다. B=25~50이 최적이 되는
파라미터 조합은 없다 — 지금 FREE=40은 어느 시나리오에서도 손해다.

## 남은 실측 3개 (실제 번역 1회의 로그로 전부 나옴)

`[gemini] model=... thinking=MINIMAL prompt=X cached=0 thoughts=Y output=Z` 로그에서:

1. **t_out** = Z ÷ 청크 블록 수
2. **th** = Y (그리고 B를 바꿔 두 번 돌리면 th가 상수인지 B비례인지 판별 — 상수면 위 결론 유지)
3. **v** = Z ÷ (청크 소요초 − TTFT), TTFT는 첫 SSE 이벤트까지 시간

추가로 정할 것 (제품 결정, 계산 불가):
- **손실 허용치**: 청크 하나가 통으로 실패하면 B블록이 원문(영어)으로 남는다.
  B=150이면 영화의 18%다. 이 최악 케이스를 어디까지 허용할지가 B의 상한.
- 진행 링 해상도: B=200이면 851블록 영화가 5청크 → 진행률이 20%씩 점프.
