# 청크 크기 수학모델

`scripts/chunk-model.mjs`가 이 모델의 실행본이다. 기본값은 **전부 실측치**로 갈아끼워져 있다.

```bash
node scripts/chunk-model.mjs                                  # 실측 파라미터
node scripts/chunk-model.mjs N=1500                           # 크레딧 상한 파일
node scripts/chunk-model.mjs file=samples/subtitles/full-movie.srt
```

> **결론 먼저**: `SERVER_CHUNK_SIZE = 125`, `SERVER_CONCURRENCY = 14`.
> 근거는 아래 5절. 무료 티어(150/6)는 **현재 미사용**이다 — BYOK가 삭제돼서
> 전원이 server 티어를 쓴다(`resolveTier()`는 항상 `'server'`).

---

## 1. 실측 파라미터 (2026-07-21)

461블록 파일 1회 번역의 `[gemini]` 로그 3줄에서 전부 나왔다. 로그 형식:

```
[gemini] model=... thinking=LOW prompt=P cached=0 thoughts=T output=O
```

| 관측 | 블록 | prompt | output |
|---|---|---|---|
| 청크 3 | 61 | 1,942 | 932 |
| 청크 1 | 200 | 3,961 | 3,165 |
| 청크 2 | 200 | 4,090 | 3,383 |

`prompt = P_fixed + blocks × t_in` 연립으로 역산:

| 기호 | 뜻 | **실측** | 이전 추정 | 추정 대비 | 비고 |
|---|---|---|---|---|---|
| t_in | 블록당 입력 토큰 | **15** | 10 | +50% | |
| t_out | 블록당 출력 토큰 | **16** | 20 | −20% | |
| **th** | 요청당 thinking 토큰 | **0** | 100 | −100% | MINIMAL·LOW 모두 0 (gemini-limits §6) |
| P_fixed | 청크마다 반복되는 고정 프롬프트 | **1,028** | 700 | +47% | 작품 notes 포함 |
| v | 출력 생성 속도 | **~220 tok/s** | 120 | +83% | 동시 시작 청크들의 완료시각 차로 역산 (175~244 범위) |
| dens | 최밀 구간/평균 밀도비 (p95) | 1.25 | — | — | 미갱신 (샘플 실측 ~1.2) |
| TTFT | 첫 토큰까지 지연 | 2s (추정) | — | — | 미측정 |

추정이 전부 한 방향으로 틀린 게 아니라는 점이 중요하다 — 입력은 과소평가(t_in +50%,
P_fixed +47%), 출력은 과대평가(t_out −20%, th −100%)였다. 비용 결론이 뒤집힌 건
**출력 쪽 과대평가 때문**이다(3절).

부수 실측: **정렬 실패율 0.22%** (461블록 중 1블록이 원문 유지 — 실패는 청크 1에만
발생, 그 청크 안에서는 1/200 = 0.5%). 이전 측정은 461블록 중 3블록 = 0.65%였으므로
**이번이 더 낮다.** 표본 2개뿐이라 0.2~0.7% 범위로 보는 게 안전하다.

## 2. 공식

```
비용  C(B) = [m·P_fixed + N·t_in]·pin  +  [N·t_out + m·th]·pout
              └ 입력: 반복 프롬프트만 B에 종속 ┘   └ 출력: thinking만 B에 종속 ┘

청크 소요  D(B) = TTFT + (B·t_out + th) / v

동시성     K = min( m, kmax )          ← 4절 참조. RPM 항은 유료에서 비바인딩
총 시간    T(B) = ⌈m/K⌉ · D            (m = ⌈N/B⌉)

제약  B·t_out·dens + th ≤ outcap        (최밀 청크가 출력 상한 안)
      D(B) ≤ timeout                     (청크 하나가 300초 안)
      B ≤ 손실 허용치                     (청크 통실패 = B블록 원문 잔존)
```

---

## 3. th=0이 결론을 뒤집었다

예전 모델의 핵심 주장은 **"비용은 B에 단조 감소하니 되도록 크게"**였고, 그 힘은 전부
`m·th·pout` 항에서 나왔다 — 요청마다 붙는 thinking이 **출력 단가(입력의 6배)**로 과금되니
요청 수(m)를 줄여야 한다는 논리였다.

**`th=0`으로 실측되면서 그 항이 사라졌다.** 남은 B 종속 항은 `m·P_fixed·pin` 하나뿐이고,
입력 단가는 출력의 1/6이라 영향이 작다. 그 결과:

| B | $/편 (N=851) | 총 시간 (K=14) | 최악 손실 |
|---|---|---|---|
| 100 | $0.1556 | 9초 | 100블록 (12%) |
| **125** | **$0.1525** | **11초** | **125블록 (15%)** |
| 150 | $0.1509 | 13초 | 150블록 (18%) |
| 200 (구값) | $0.1494 | 17초 | 200블록 (24%) |
| 851 | $0.1432 | 64초 | 851블록 (100%) |

**B=125와 B=851의 비용 차이는 6.5%**다. 비용이 사실상 평평해진 이상, B는 이제
**시간·손실반경·진행률 해상도**로 정해야 한다 — 셋 다 작은 B를 선호한다.

## 4. K는 제약이 아니라 우리 선택이다

`gemini-limits.md` §7-1 조회 결과, **인프라 병목이라는 전제가 틀렸다**:

| 후보 | 실제 값 | 바인딩? |
|---|---|---|
| Gemini 동시 요청 제한 | 없음 (§2) | ✗ |
| Vercel 동시 실행 | 30,000 자동 스케일 (§7-1) | ✗ — 우리 14와 3자릿수 차 |
| Gemini 유료 RPM | 1,000 (전체 사용자 공유) | △ 규모가 커지면 |
| 라우트 타임아웃 300초 | B=851도 64초 | ✗ |
| 출력 상한 65,536 | B=125면 2,500 | ✗ |

즉 **K를 막는 건 아무것도 없다.** K는 "Gemini의 1,000 RPM 중 한 사용자에게 얼마를
할당할 것인가"라는 배분 결정일 뿐이다. 현재 K=14, B=125면 1인당 38 RPM →
**동시 번역 26명**까지 여유. 지금 규모에선 무의미한 제약이라 K=14를 유지한다.

## 5. 결정: B = 125

**결정 규칙: 우리가 받는 모든 파일이 1웨이브에 끝나는 가장 작은 B.**

크레딧 1개가 커버하는 상한이 `MAX_BLOCKS_PER_CREDIT = 1,500`블록이므로,
**1,500블록보다 큰 파일은 애초에 받지 않는다**(413 `file_too_large`). 따라서 최악의
경우만 만족하면 된다:

```
⌈1500 / B⌉ ≤ K = 14      →      B ≥ 1500/14 = 107.1
```

| B | N=1500일 때 청크 수 | K=14 대비 여유 | 총 시간 | 판정 |
|---|---|---|---|---|
| 100 | 15 | **초과** | 2웨이브 (19초) | ✗ 상한 파일이 2웨이브 |
| 110 | 14 | 0 | 1웨이브 (10초) | △ 하한에 딱 붙음 |
| 120 | 13 | 1 | 1웨이브 (11초) | ○ |
| **125** | **12** | **2** | **1웨이브 (11초)** | **✓ 채택** |
| 150 | 10 | 4 | 1웨이브 (13초) | 느리고 손실반경 큼 |
| 200 | 8 | 6 | 1웨이브 (17초) | 구값. 가장 느림 |

**B=125를 고른 이유** — 하한(108)을 넘는 값 중 최소는 아니다. 110·120도 1웨이브를
만족한다. 125를 고른 건 세 가지 때문이다:

1. **`1500 / 125 = 12`로 나누어떨어진다.** 상한 파일에서 마지막 청크가 잘리지 않는다.
2. **여유 2청크.** 110은 K와 정확히 같아 K를 하나만 줄여도, 또는 크레딧 상한을 조금만
   올려도 곧바로 2웨이브가 된다. 여유가 이 값을 결정에 덜 민감하게 만든다.
3. **110과 125의 실측 차이가 없다.** 둘 다 1웨이브 10~11초다. 더 작게 갈 이유가 없으면
   손실반경이 작은 쪽이 아니라 **여유가 큰 쪽**을 고르는 게 맞다 — 시간이 같으므로.

**구값 200 대비**: 영화 한 편 17초 → 11초, 손실반경 24% → 15%, 진행 링 5칸 → 7칸,
비용 +$0.0031(약 4원)/편.

### 동적 B를 채택하지 않은 이유

`B = ⌈N/K⌉`로 파일마다 맞추면 851블록에서 6.4초까지 줄지만,
- 요청 수가 7개 → 14개로 늘어 **1인당 RPM 소비가 3.4배**가 되고(38 → 131 RPM,
  동시 번역 가능 인원 26명 → 7명),
- 크레딧 상한이 1,500블록이라 **고정 B=125가 이미 전 구간을 1웨이브로 덮는다.**

4~5초를 줄이려고 동시 수용 인원을 1/4로 깎는 거래다. 복잡도까지 감안하면 이득이 없다.

## 6. 무료 티어(150/6)의 현재 지위

BYOK 삭제로 **현재 아무도 쓰지 않는다.** 상수와 `getTierLimits('free')`는 남겨뒀다:

- 값 자체는 여전히 유효하다 — Gemini 무료 등급 RPM 15에서 유도됐고 그 한도는 안 변했다
- 로그인 후 "가입은 했지만 크레딧이 없는" 티어를 만들 때 다시 쓸 자리다

`resolveTier()`가 항상 `'server'`를 반환하므로 실행 경로에는 안 들어온다.

## 7. 남은 미측정

- **THINKING_LEVEL MEDIUM/HIGH** — 품질 비교용. LOW까지 th=0인 게 확인됐으니
  MEDIUM 이상에서 th가 살아나는지, 그 비용이 품질값을 하는지가 관심사.
  절차: `.env.local`의 `THINKING_LEVEL` 변경 → **dev 서버 재시작 필수**(모듈 로드 시 1회 읽는 상수라 핫리로드 안 먹음) → 재번역 → 로그의 `thoughts=` 확인
- **TTFT** — 2초는 추정. 첫 SSE 이벤트까지 시간으로 측정 가능
- **dens** — 1.25는 이전 샘플 실측(~1.2)의 보수적 반올림. 출력 상한(65,536)에 한참
  못 미쳐서(B=125면 2,500) 정밀화 실익 없음
