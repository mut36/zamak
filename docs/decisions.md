# 결정 기록

**왜 지금 이렇게 되어 있는가.** 코드를 읽으면 아는 "무엇"이 아니라, 코드에 안 남는
"왜"와 "무엇을 버렸는가"를 적는다. 뒤집힌 결정은 지우지 않고 취소선으로 남긴다 —
같은 논의를 다시 하지 않기 위해서다.

튜닝 수치(B/K)의 상세 유도는 [`tuning/chunk-size-model.md`](tuning/chunk-size-model.md),
API 한도 원본은 [`tuning/gemini-limits.md`](tuning/gemini-limits.md).

## 1. 과금·접근 모델

### 1-1. BYOK(사용자 API 키)는 완전히 삭제됐다 — 2026-07-21

세 번 뒤집힌 끝의 최종 상태다.

| 시점 | 상태 |
|---|---|
| ~2026-07-18 | 하이브리드 — 키를 넣으면 그 키로, 없으면 서버 키로 |
| 2026-07-18 | ~~BYOK 필수~~ — 결제가 없으니 전원이 자기 키를 내라 |
| 2026-07-21 | ~~BYOK 선택~~ — 서버 키를 기본값으로, 키는 옵션 |
| **2026-07-21** | **BYOK 삭제** — 사용자가 API 키를 보는 화면 자체가 없다 |

**이유**: 어차피 로그인 + 서버 키 + 크레딧으로 가기로 확정된 이상, BYOK는 언젠가
지울 코드였다. 그리고 결정적으로 — **무료 Gemini 키는 Google Search grounding을
아예 지원하지 않는다.** grounding은 `/api/enrich`가 작품의 감독·톤·인물 말투를
가져오는 수단이고, 이건 우리 제품의 방어선 2개 중 하나다. BYOK 사용자는 구조적으로
이 기능을 못 쓴다 — 즉 BYOK 티어는 "느린 무료"가 아니라 "핵심 기능이 빠진 무료"였다.

**삭제된 것**: 키 입력 UI, `geminiKey` state·sessionStorage, `x-gemini-key` 헤더
(클라·서버 양쪽), `apiKeyError.ts`, `ApiKeyField.tsx`, invalid-key UX 전체.

**남긴 것**: provider 레이어의 `apiKey` 파라미터. 계정별 키가 생길 경우의 seam이고
아무 비용이 없다.

> 교훈: 기획이 죽인 기능을 "옵션"으로 반쯤 남기지 말 것. 한 번 그렇게 했다가
> 아무도 채울 이유가 없는 입력란만 화면에 남았다.

### 1-2. 익명 미리보기는 폐기됐다 — 2026-07-21

앞 50줄을 서버 키로 무료 번역해 보여주고 그 아래에 페이월을 두는 방식. 구현·검증까지
마쳤으나 **당일 폐기**했다.

**이유**: 막을 수 없는 어뷰징 벡터. 같은 파일을 51~100줄, 101~150줄로 잘라 "다른 파일"인
척 반복 요청하면 IP당 일일 제한 안에서도 전체를 공짜로 뽑아간다. 레이트리밋은 요청 횟수만
세지 파일 내용을 보지 않는다. 파일 해시 추적으로 막으려 하면 복잡도가 계속 붙는데,
**더 단순한 해법은 "익명이 서버 키를 쓰는 엔드포인트를 아예 없애는 것"**이었다.

대체안이 아래 1-3이다. 코드는 전량 제거됐다.

### 1-3. 로그인 게이트 + 선불 크레딧 — 2026-07-21 구현 완료

```
방문 → 로그인 벽 → Google 로그인 → 가입 즉시 크레딧 1개 → 전체 번역 1회 무료
     → 이후 크레딧 구매 (결제 미구현, "준비 중"에서 멈춤)
```

- **어뷰징 표면이 사라진다.** 익명이 서버 키를 쓰는 경로가 없고, 계정은 구글이
  사실상 신원 확인을 대신해준다.
- **완결된 결과물이 미완성 미리보기보다 강한 미끼.** "손에 쥐어본 사람만 두 번째에
  결제한다"는 논지는 유지하되, 50줄 샘플 대신 진짜 완성품을 준다.
- 다계정으로 무료 1편을 반복 수령하는 건 이론상 가능하지만 일반적인 SaaS 프리미엄
  어뷰징 수준이라 우선순위가 낮다. 익명 API 자금 유출과는 리스크 등급이 다르다.

### 1-4. 크레딧은 요청이 아니라 **파일** 단위로 차감한다

**이것이 이 시스템에서 가장 중요한 설계 결정이다.**

영화 한 편은 청크 여러 개로 쪼개져 `/api/translate`를 N번 호출한다. 요청마다 차감하면
한 편에 크레딧이 7~12개 날아간다. 그래서:

1. `/api/translation/begin`이 **크레딧 1개 차감 + job 생성을 하나의 SQL 함수
   (`begin_translation_job`)에서** 수행한다 — 단일 statement의 행 잠금 덕에 탭 두 개가
   마지막 크레딧을 동시에 쓸 수 없다.
2. 이후 모든 청크 요청은 `jobId`를 함께 보내고, 서버가 소유권·유효기간을 검증한다.

이게 없으면 `/api/translate`는 "로그인만 하면 무제한"이 된다. 크레딧이 begin에서
차감되므로, **그 지불을 증명하는 자리가 청크 엔드포인트**다.

*실측 검증(2026-07-21)*: 461블록 → 청크 3개 → Gemini 3회 호출 → `translation_jobs` 행
1개, 잔액 1 → 0. 설계대로 동작한다.

### 1-5. 보조 라우트는 로그인만 요구하고 크레딧은 안 깎는다

`/api/analyze`, `/api/enrich`, `/api/summarize`는 값싼 flash-lite 호출이다. 사용자가
결과를 보고 버릴 수도 있는 메타데이터에 과금하는 건 부당하고, 금액도 무의미하다.
다만 **서버 키를 쓰므로 로그인은 반드시 요구한다.**

### 1-6. 설정 누락 시 fail-closed

Supabase 설정이 없으면 모델 라우트는 **500으로 닫힌다.** 열어두고 넘어가면 설정 실수가
곧 "인증 없는 서버 키 개방"이 되는데, 그건 이 게이트가 막으려는 바로 그 상태다.

### 1-7. 미해결 — 번역 실패 시 크레딧 환불 없음

크레딧은 begin에서 차감되므로 이후 번역이 실패해도 돌려주지 않는다. 부분 실패는 실패한
청크가 원문을 유지해 **재생 가능한 완전한 SRT**가 나오므로 결과물은 손에 남는다는
판단으로 수용했다. 실패율이 오르면 재검토 대상.

### 1-8. 랜딩은 벽 앞의 정적 콘텐츠, 단일 스크롤, `/` 유지 — 2026-07-21

로그인 벽은 내릴 수 없으므로(모델 라우트 전부 서버 키) 랜딩의 유일한 일은 "로그인
버튼을 누를 이유"를 만드는 것이다. 전부 정적 콘텐츠라 API 비용 0.

- **구조**: `app/page.tsx`의 `!user` 분기에서 `LandingPage` 렌더. `/`·`/app` 분리는
  검토 후 보류 — 결제가 붙어도 필요한 건 `/terms`·`/refund` 같은 문서 페이지지
  라우트 분리가 아니고, 분리 비용(OAuth 콜백 수정)은 지금 낼 이유가 없다.
  `LandingPage`는 독립 컴포넌트라 분리가 필요해지면 import 한 줄만 옮기면 된다.
- **메뉴바 없음**: 이동할 페이지가 없는데 메뉴를 두면 빈 약속. 대신 푸터를 미리
  두어 결제 날 법적 고지·가격 링크가 들어갈 자리를 확보했다.
- **가격 미노출** (대표 위임 후 결정): 잠정 가격은 원가 실측 후 조정될 수 있어
  숫자를 박지 않는다. 대신 "사람에게 맡기면 편당 15만 원" 앵커 + 무료 1편만 노출.
- **브랜드 톤 = 도구, 타깃 = 국내 우선·해외는 나중** (대표 확인). 카피는 한국어
  한 벌, `COPY.landing` 네임스페이스.
- **샘플은 자체 제작 예문** (형사물 선배/신입 대화 3블록) — 저작권 0. 방어선 2개
  (타임코드 동일, 관계 기반 존댓말/반말)가 눈으로 비교되도록 설계.
- 인터랙티브 데모(가짜 위저드 시뮬레이션)는 각하 — 구현 비용 대비, "실제가
  아님"이 드러나는 순간의 신뢰 역효과가 더 크다.

## 2. 번역 파이프라인

### 2-1. 타임코드는 코드가 관리하고 AI는 대사만 다룬다

모델에는 **번호 + 대사만** 보낸다(타임스탬프는 토큰 낭비). 응답은 **번호로 대조해**
원본 타임코드와 재결합한다(`reassembleTranslatedChunk`).

AI가 자막을 합치거나 빠뜨려도 **이후 자막이 밀리는 일이 구조적으로 불가능하다** —
타임코드의 출처가 항상 원본이기 때문이다. API 추가 호출도 0이다.

번호 인정 조건 3개(원본에 있던 번호 / 아직 미사용 / 단조 증가)로 `1999` 같은 숫자
대사를 번호로 오인하지 않는다. 인정 실패 시 본문에 섞지 않고 버려서 그 블록만 원문 폴백.

*실측 정렬 실패율 0.5~0.65%.*

### 2-2. 기본 경로는 청크당 모델 호출 1회 — 검증·재시도 없음

과거 `translationService`가 블록 수 검증 실패 시 200블록 청크를 개별 블록 200번
재귀 번역 + 3회 재시도까지 해서 **20분/5,000원 사고**를 냈다. 지금은 청크당 1회 호출
+ raw 반환이 기본이고, 실패 청크는 클라이언트가 원문 그대로 유지한다.

검증·재시도·재귀 로직은 삭제하지 않고 `TRANSLATION_STRICT_MODE=true`(기본 off) 뒤에
보존했다.

### 2-3. 청크 크기 B=125, 동시성 K=14

유도 전체는 [`tuning/chunk-size-model.md`](tuning/chunk-size-model.md). 요지만:

- **thinking 토큰이 실측 0**이라 비용 곡선이 평평해졌다(B=125 vs 851이 6% 차이).
  "클수록 싸다"는 옛 논거가 사라졌다.
- 그래서 B는 **시간·손실반경·진행률 해상도**로 정한다 — 셋 다 작은 B를 선호.
- 하한은 크레딧 상한에서 나온다: `⌈1500/14⌉ = 108` 이상이어야 우리가 받는 모든 파일이
  1웨이브에 끝난다. **125가 이를 만족하는 가장 작은 실용값**(여유 2청크).
- K는 제약이 아니라 배분 선택이다. Gemini도 Vercel도 동시성을 막지 않는다.

### 2-4. THINKING_LEVEL은 LOW

`thinkingBudget: 0`은 이 모델이 무시한다(비활성화 불가). 실제로 먹는 건 `thinkingLevel`.

**MINIMAL과 LOW 모두 실사용에서 `thoughts=0`**으로 측정됐다 — 비용이 같으므로 품질이
나은 LOW를 **코드 기본값으로** 삼았다(`constants.ts`). env로 덮을 수 있다.

MEDIUM/HIGH는 미측정이다. **측정 전까지는 thinking 토큰을 쓴다고 가정할 것** — 쓴다면
출력 단가(입력의 6배)로 요청마다 붙으므로 비용 성격이 LOW와 완전히 다르다.

## 3. 검토 후 각하한 대안들

| 대안 | 각하 이유 |
|---|---|
| `{id,text}` JSON Schema 강제 | 출력 토큰이 더 비싸고, **스키마는 문법만 강제하지 의미적 1:1 대응을 보장 못 한다** |
| 전체 자막 입력 + 출력만 청킹 | 캐싱은 과금·지연 최적화지 **어텐션 최적화가 아니다.** 시프팅을 못 줄이고 "구간 찾기" 오류가 추가됨 |
| 2회 돌려 교차검증 | 비용 2배 |
| 동적 B (`⌈N/K⌉`) | 고정 B=125가 이미 전 구간을 1웨이브로 덮는다. 복잡도 대비 이득 없음 |
| 구독 우선 | 자막 번역은 "간헐적 폭발형" 수요(몇 달 안 쓰다 하룻밤에 3편)라 안 쓴 달에 해지당함 |
| 평생 라이선스 | 원가가 사용량 비례라 자살행위 |
| 건별 종량(매번 카드) | 매번 결제창 = 매번 이탈 |
| 에이전시 B2B | 1인 개발 리소스로 6~12개월 세일즈 사이클 감당 불가 |
